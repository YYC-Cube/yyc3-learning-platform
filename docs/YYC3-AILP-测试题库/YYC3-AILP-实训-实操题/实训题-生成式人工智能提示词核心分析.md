# 生成式人工智能应用工程师（高级）实训题-提示词核心解析

- 提示词工程师， 深入解析生成式人工智能提示词设计的核心定义。
- 生成式AI提示词设计的核心，可以定义为：
  - 通过精心构建的文本指令，
  - 引导模型进入一个特定的、
  - 受限的“思维情境”，从而精准、可靠地激发模型产生符合预期目标的输出

🌹 生成式提示词设计的五大核心定义 🌹

## 情境构建 - 设定思维的舞台

**这是最基础也是最重要的一步。你需要为AI设定一个角色、一个场景和一个任务目标**
• 核心作用：告诉模型“它现在是谁”、“在什么环境下”、“要做什么”。这极大地限制了模型漫无边际的联想，将其知识库聚焦于一个特定领域。
• 好比：你不能只对演员说“表演”，而要说“你是一位在雨夜失去孩子的父亲，表演出他的绝望”。

## 任务定义 - 明确输出的靶心

**任务必须具体、可操作、无歧义。它直接决定了输出结果的形态**
• 核心作用：明确告诉模型要产出的“具体产物”是什么，是列表、文章、代码、分析报告，还是对话回复。
• 关键要素：包含动作（如：分析、总结、生成、分类、翻译）和对象（如：评论数据、营销文案、Python代码）。

## 约束与规格 - 划定思维的边界

**没有约束的模型会天马行空。通过设定规则，确保输出的可控性和可用性**
• 核心作用：从格式、内容、风格、长度等方面对输出进行限制，确保结果符合后续处理或展示的要求。
• 例如：规定输出为JSON格式、字数不超过300字、避免使用专业术语、必须包含某些特定要素等。

## 示例引导 - 提供思维的范本

**对于复杂任务，说一千道一万，不如给个例子看。少样本学习是引导模型最有效的方式之一**
• 核心作用：通过提供“输入-输出”对，让模型通过类比来理解任务的细微要求，特别是对于风格、语气、逻辑结构等难以用语言精确描述的特征。
• 效果：能极大提高输出结果的稳定性和质量，是实现“精益”输出的关键。

## 迭代优化 - 持续校准的过程

**提示词设计很少一蹴而就。它是一个基于模型反馈进行持续调试和优化的动态过程**
• 核心作用：通过分析不理想的输出，反推提示词在哪一个核心点上存在模糊或缺失，并进行针对性修正。
• 思维模式：将模型的错误输出视为“提示词本身的bug”，而不是“模型的失败”。

🚀 核心总结

**我们可以用一个精炼的公式来概括提示词设计的核心：**

- 卓越的提示词 = 精准的情境 + 明确的任务 + 严格的约束 + （可选的示例）

**在提示词评论分析中，完美贯彻这五大核心的典范**
• 情境：设定为“智能意图识别系统”。
• 任务：明确为“判断意图类别”并“撰写分类理由”。
• 约束：定义了四种类别、给出了输出格式、限定了理由长度。
• 示例：提供了正例来对齐风格和逻辑。
• 迭代：题干本身的要求就是对提示词不断优化的过程。
**掌握这五大核心，您就掌握了与生成式AI高效、精准沟通的根本法则**

## 用户评论数据智能问数

你的任务是实现一个用户评论数据的智能问数，用户评论表mysql数据库的表结构如下，请你设计一个prompt能实现根据“样例数据”中的⾃然语⾔提问⽣成对应的预期结果的sql语句

-- usergoods 商品表

CREATE TABLE usergoods (

    product_id INT NOT NULL AUTO_INCREMENT COMMENT '商品ID',

    product_name VARCHAR(255) NOT NULL COMMENT '商品名称',

    product_description TEXT COMMENT '商品描述',

    product_price DECIMAL(10, 2) NOT NULL COMMENT '商品价格',

    PRIMARY KEY (product_id)

);

-- usercomments 用户商品评价表结构

CREATE TABLE usercomments (

    user_id INT NOT NULL COMMENT '用户ID',

    product_id INT NOT NULL COMMENT '商品ID',

    comment_content TEXT NOT NULL COMMENT '评价内容',

    comment_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '评价时间',

    comment_score TINYINT CHECK (comment_score BETWEEN 1 AND 5) NOT NULL COMMENT '评价分数（1-5）3分以下是差评，3分以上是好评',

    PRIMARY KEY (user_id, product_id),

    FOREIGN KEY (product_id) REFERENCES usergoods(product_id) ON DELETE CASCADE ON UPDATE CASCADE

);

要求

（1）需要解决大模型的幻读问题，防止“好评，差评”等语义输入大模型SQL查询条件输出的结果不对

（2）下⾯参考输⼊内容中的不同输⼊都需要能正确执⾏prompt并输出与预期结果同等查询逻辑的SQL 语句

样例

1. input: 2024-11-26日这天商品“拖鞋”的好评率是多少

output:

SELECT

    (SUM(CASE WHEN uc.comment_score >= 3 THEN 1 ELSE 0 END) / COUNT(*)) * 100 AS good_rate

FROM

    usercomments uc

JOIN

    usergoods ug ON uc.product_id = ug.product_id

WHERE

    DATE(uc.comment_time) = '2024-11-26'

    AND ug.product_name = '拖鞋';

2. input:统计12月份商品“毛衣”的差评数量

output:

SELECT COUNT(*) AS bad_comment_count

FROM usercomments uc

JOIN usergoods ug ON uc.product_id = ug.product_id

WHERE MONTH(uc.comment_time) = 12

AND ug.product_name = '毛衣'

AND uc.comment_score < 3;

1. input:找到商品好评数量最大的前100用户

output:

SELECT user_id, COUNT(*) AS good_comment_count FROM usercomments WHERE comment_score >= 3 GROUP BY user_id ORDER BY good_comment_count DESC LIMIT 100;
